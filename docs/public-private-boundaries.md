# Public/Private Boundaries

## Purpose

This document defines what may appear in public safety notes and what must remain private or sealed.

## Boundary Classes

| Class | Definition | Handling |
| --- | --- | --- |
| Public | Already reviewed for publication and safe to link or summarize. | May appear with status and source. |
| Private | Authorized operating-group material. | Do not publish or summarize without approval. |
| Sealed | Source, artifacts, methods, or assumptions requiring explicit human authorization before excerpt, mirror, release, or derivative summary. | Do not include. |
| Mixed | Contains both public and non-public material. | Split by path, branch, artifact, or repository before publication. |

## Public-Safe Content

- standards;
- checklists;
- synthetic examples;
- generic limitation language;
- public status labels;
- links to public reviewed artifacts.

## Forbidden Public Content

- donor data;
- student data;
- school private data;
- volunteer data;
- private Foundation operations;
- customer data;
- partner private records;
- private training corpora;
- model weights;
- dataset files or unapproved sanitized samples;
- private evaluations;
- private telemetry;
- production prompts;
- sealed YOSO-YAi LLC IP;
- secrets, tokens, and API keys;
- exact sensitive infrastructure locations;
- security-sensitive NEURONA operational details.

## Release Claims

A public artifact may say `released` only when the public artifact exists, the review state is documented, and human publication authority has approved the claim.
